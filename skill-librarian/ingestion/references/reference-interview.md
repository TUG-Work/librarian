# The Reference Interview

A practice borrowed from library science. The reference librarian knows that **the first question asked is rarely the actual need**.

## Why This Matters

When someone says "organize these 500 documents into categories," they're stating a task. But tasks emerge from needs, and needs emerge from situations:

```
Situation → Need → Task → Question Asked
```

A skilled reference interview works backward:
- **Question asked:** "Organize these documents"
- **Task:** Create categories for navigation
- **Need:** Users can't find what they need
- **Situation:** Website redesign, content migration, compliance audit, etc.

The *situation* determines what kind of organization actually helps.

## The Constraint

Reference librarians can seem like gatekeepers. Ask one or two questions max, then move forward. Be helpful, not an intake form.

## Core Questions

### Question 1: Purpose

> "What are you trying to accomplish with this corpus?"

Listen for:
- **Navigation** → They need browse structures, categories
- **Migration** → They need mapping, gap analysis
- **Governance** → They need ownership, lifecycle metadata
- **Compliance** → They need classification against an external standard
- **Search improvement** → They need vocabulary, synonyms, tagging
- **Content audit** → They need inventory, gap analysis, quality assessment
- **Research** → They need themes, patterns, relationships

### Question 2: Audience (if not obvious)

> "Who will use the results of this work?"

Listen for:
- **End users** → Focus on findability, comprehension
- **Content authors** → Focus on templates, guidelines, governance
- **Developers** → Focus on structure, schema, API contracts
- **Executives** → Focus on high-level themes, metrics
- **Compliance officers** → Focus on completeness, audit trails

### Question 3: Constraints (if time permits)

> "Are there existing categories, taxonomies, or structures we need to work with?"

This surfaces:
- External standards that must be adopted
- Legacy structures that can't change
- Political constraints on categorization

## What the Answers Change

| Purpose | Analytical Emphasis |
|---------|---------------------|
| Navigation | Shallow broad categories, user mental models |
| Migration | Mapping to target structure, gap identification |
| Governance | Ownership facets, lifecycle metadata, retention |
| Compliance | Classification against external taxonomy |
| Search | Vocabulary extraction, synonym rings, controlled terms |
| Content audit | Inventory, quality flags, gap analysis |
| Research | Deep topic modeling, relationship mapping |

## When to Skip the Interview

- **Clear mandate exists:** Someone already defined the goal in writing
- **Exploratory engagement:** They genuinely want to see what's there first
- **Technical request:** They just need corpus stats, not advice

In these cases, proceed to profiling and let the Capabilities Briefing surface options.

## Anti-Patterns

1. **Interrogation:** More than 2-3 questions feels like a form
2. **Assumptions:** Don't assume "organize" means "categorize"
3. **Jargon:** Don't ask about "faceted classification" — ask what they're trying to do
4. **Gatekeeping:** Don't refuse to proceed without answers

## Example Dialogues

### Good

> **User:** I need to organize these 500 policy documents.
>
> **Agent:** Got it — 500 policy docs. Quick question: is this for helping people find policies, or for something else like a migration or audit?
>
> **User:** It's for the intranet redesign. People can't find what they need.
>
> **Agent:** Navigation use case. I'll profile the collection and show you what category structures might work. Let me start by looking at what's there.

### Also Good

> **User:** Here are 2,000 customer support tickets. Find patterns.
>
> **Agent:** On it. I'll profile these and run topic modeling to surface the main themes. Any particular patterns you're hoping to find, or is this exploratory?
>
> **User:** Exploratory. Just want to see what's there.
>
> **Agent:** Perfect. I'll show you what the data reveals.

### Too Much

> **User:** Organize these documents.
>
> **Agent:** Before I can help, I need to understand your goals, your audience, your constraints, your timeline, your success metrics, your organizational context...
>
> *(User has already left)*

## Output

The reference interview produces:
1. **Stated purpose** — what they said they want
2. **Inferred need** — what problem they're solving
3. **Constraints** — what they have to work with
4. **Audience** — who will use the output

This context shapes the Capabilities Briefing and the recommendations in the Collection Record.
